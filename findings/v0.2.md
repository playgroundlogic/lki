# LKI Format Findings — accumulating toward v0.3

**Source format version:** 0.2
**Status:** open; will close when v0.3 ships
**Opened:** 2026-05-19
**Predecessor:** [FINDINGS_v0.1.md](../v0.1/FINDINGS.md) (closed)

## Purpose

Record format-level issues surfaced through projection exercises,
spec drafting, deployment, research, and migration during the v0.2
era. Each finding documents the issue, why it matters, and a proposed
direction for v0.3. Proposals are not commitments; v0.3 may handle
differently.

new tag values compared to v0.1:

## From original v0.2 scoping

### F-034: Compute capability breach handling
**Source:** `[scoping]`
**Description:** v0.2's compute capability supports bound declarations
(time, memory, cpu, tokens, api_calls, gpu_time, etc.) but breach
handling is binary — PDP returns Deny when policy forbids the
consumption level. Real cost-aware deployments need richer breach
semantics: soft bounds (warn but proceed), approval-required bounds,
budget-pool deduction, throttling, fallback-to-cheaper-alternative.

**Why it matters:** Cost-aware agent operations are a real production
need. Anthropic API costs, third-party service call costs, GPU
allocation budgets, and cumulative resource consumption all need
policies more nuanced than "deny when exceeded."

**Proposed direction:** v0.3 adds a `breach_handling:` field on
compute capabilities:

```yaml
compute:
  - resource: tokens
    bound: 10000
    cost_model: per_unit_with_provider_pricing
    provider: anthropic_api
    breach_handling:
      strict_bound: 100000
      soft_bound_actions:
        - request_approval
        - log_warning
        - deduct_from_budget_pool
        - throttle_until_budget_available
        - degrade_to_alternative
```

This requires substantial v0.3 design work:
- Budget pool entities in Cedar with stateful tracking
- Approval workflow integration with agent runtime
- Cedar conditional-permit semantics (Allow with conditions the
  resolver must satisfy)
- Audit log records of breach decisions and their resolutions

Designed incrementally — different breach handling actions can ship
in different v0.x revisions if the full set proves to be too much for
v0.3.

## From v0.1 → v0.2 migration work

### F-035: Shell built-ins as Tool entities
**Source:** `[migration]`
**Description:** v0.2's Tool registry requires every tool referenced
by `process.spawn` capabilities or `implementation.tool` fields to
exist in the registry. This worked cleanly for migrations of fetch_url
(curl, wget), grep_files (rg, grep), etc., but surfaced an ambiguity
for write_file: its template uses `printf`, `mktemp`, `mv`, `chmod`,
`sync` — some are shell built-ins, some are coreutils binaries. v0.2
implicitly treats all of these as needing Tool entities.

**Why it matters:**
1. Shell built-ins (`printf`, `cd`, `set`, etc.) aren't separately
   invoked processes — they're part of the shell itself. Treating
   them as Tool entities is semantically odd.
2. The Process::Spawn capability for `printf` doesn't make sense
   because printf doesn't spawn a new process when invoked from a
   shell context.
3. Conversely, treating shell built-ins as "not tools" creates a
   special case in the format that complicates tooling.

**Proposed direction:** v0.3 distinguishes between:
- **Spawnable tools** — separately-invoked binaries (curl, git, rm,
  cat, find). These get Tool entities in the registry. Process::Spawn
  capability applies.
- **Shell builtins** — built into the shell, not separately invokable
  (printf, cd, set, test, [, [[, while, for). These don't get Tool
  entities. A new `shell_builtin: true` annotation in implementations
  signals that the template uses builtins; no Process::Spawn
  capability needed for them.

A new `shell` capability category may be introduced for
shell-mediated operations as a category distinct from process.spawn.

### F-038: FilePath observed attributes incomplete for common policy needs
**Source:** `[migration]`
**Description:** v0.2's FilePath@1.x type declares `is_system_path` as
an observed attribute covering /etc, /usr, /bin, /sbin, /var, /boot,
/lib, /lib64 trees. read_file's v2.0.0 migration surfaced that real
anti-patterns reference categories beyond this set:
- `/proc/*` and `/sys/*` (pseudo-filesystems with privilege-relevant
  content)
- `/dev/*` (device files including character devices like /dev/urandom)
- `/home/<other-user>/*` (other-user paths, distinct from
  is_within_working_dir negation)
- `/tmp/*` and `/var/tmp/*` (shared-tmp paths with potential cross-user
  visibility)

Without observed attributes for these categories, the anti-patterns
must hand-craft Cedar policies referencing specific path prefixes,
which is less robust than referencing typed attributes on FilePath.

**Why it matters:** Three failure modes:
1. Inconsistent policy coverage — different specs handle these paths
   differently because there's no shared attribute to reference
2. Tenant policy template proliferation — every deployment hand-rolls
   the same path-prefix checks
3. Type-catalog drift — different LKI implementations may extend
   FilePath with different ad-hoc attributes if the canonical type
   doesn't cover real needs

**Proposed direction:** v0.3 extends FilePath@1.x → FilePath@1.1.0
(minor version bump, additive) with additional observed attributes:

```yaml
observed_attributes:
  is_system_path:           # existing
    type: bool
  is_pseudo_filesystem:     # new in @1.1.0
    type: bool
    description: "True for /proc, /sys paths"
  is_device:                # new in @1.1.0
    type: bool
    description: "True for /dev paths"
  is_user_home_other:       # new in @1.1.0
    type: bool
    description: "True when path is in /home/<user> for a user other than principal"
  is_shared_tmp:            # new in @1.1.0
    type: bool
    description: "True for /tmp, /var/tmp paths"
```

Specs that don't reference these attributes continue working with the
@1.x pin. Specs that want the new attributes update to @1.1.x.

Open question: should this be a v0.3 format bump or just a type
registry minor version bump under v0.2? The latter is technically
non-breaking (additive only) and could ship without waiting for v0.3.
Decision deferred; for now F-038 tracks the need.

### F-037: Map iteration in interpolation grammar
**Source:** `[migration]`
**Description:** v0.2's interpolation grammar supports list-flatten
(`{name[separator]}`) but has no equivalent for map iteration. The
fetch_url v2.0.0 migration surfaced this: rendering `headers` (a
`map<string,string>`) as repeated `--header 'Name: Value'` flag pairs
isn't expressible in v0.2's grammar. The migration worked around this
by declaring `resolver_expansion: "map_to_header_flags"` in
parameter_mapping, but this is an informal marker — the v0.2 grammar
doesn't actually define resolver_expansion as a valid parameter_mapping
field.

**Why it matters:** Map-typed parameters appear in many real intents.
HTTP headers, environment variables for process spawn, query parameters,
database connection parameters, container labels, tags on cloud
resources — all are maps that need iteration for command rendering.
Without grammar support, each spec invents its own informal escape
hatch (as fetch_url just did) or every map-based template requires
custom resolver-side code per intent.

**Proposed direction:** v0.3 extends the interpolation grammar with
map iteration syntax. One option:

```
{name{template_per_entry, separator}}
```

For example:
```
{headers{--header '$key: $value', }}
```

Iterates over headers, producing `--header 'Name: Value'` per entry,
joined by space. The `$key` and `$value` are special interpolations
valid only inside the per-entry template.

Alternative: a render modifier with a templating sub-syntax:
```
{headers|map_template:--header '$key: $value':, }
```

Both are awkward. The right design needs more thought, but the
need is clear: v0.3 must support map iteration as a first-class
grammar feature.

### F-036: Recursion in conditional cases for capability declarations
**Source:** `[migration]`
**Description:** When migrating write_file's three-axis state_mutation
to v0.2, the three axes' values *depend on parameters* (atomic and
fsync). v0.2's conditional cases mechanism works for templates but
isn't fully specified for capability declarations with internal
structure.

For state_mutation specifically, you need either:
1. Multiple state_mutation entries with `required_if:` distinguishing
   them (verbose, repeats the operation/resource_type/resource_identifier)
2. Conditional fields *within* a single state_mutation entry (not
   currently supported in v0.2 grammar)

**Why it matters:** Real intents have parameter-dependent capability
semantics. write_file is the canonical case but won't be the only
one. database_query intents will have isolation levels that vary with
parameters. send_email intents will have delivery semantics that vary.

**Proposed direction:** v0.3 extends the conditional cases mechanism
to capability declarations, with `cases:` and `when:` clauses
selecting among capability field values:

```yaml
state_mutation:
  - operation: write
    resource_type: FilePath@1.x
    resource_identifier: "{path.canonical}"
    cases:
      - when: "atomic == true and fsync == true"
        visibility: rename_atomic
        durability: fsynced
        reversibility: irreversible
      - when: "atomic == true and fsync == false"
        visibility: rename_atomic
        durability: committed
        reversibility: irreversible
      - default:
        visibility: immediate
        durability: in_memory
        reversibility: irreversible
    purpose: user_facing
```

This unifies conditional declarations across capabilities, templates,
and parameter mappings.

## Pending — not yet observed but expected

These are anticipated findings carried forward from v0.1's pending
list. They haven't surfaced in v0.2 specs yet but remain likely as
the spec library grows.

### P-002 (carried forward): Authentication context propagation
Expected from `clone_repository` and other intents that need credentials
without exposing them in parameters. SSH keys, OAuth tokens, etc.
v0.2 doesn't model how credentials flow from the principal context to
the implementation without appearing in audit logs or being treated
as ordinary parameters.

### P-003 (carried forward): Process tree spawning vs single process
Expected from `run_container` where the container itself spawns
processes. v0.2's `process.spawn` capability only covers the direct
child. A container or supervisor process that spawns other processes
needs a more nuanced capability declaration.

### P-004 (carried forward): Network policy with allowlist/denylist semantics
Expected from any spec that needs to allow some hosts but not others
based on policy. v0.2 capability declarations are descriptive (what
the intent needs) rather than prescriptive (what policy permits).
Cedar policies handle this via separate forbid/permit rules; the
declaration side could express it more directly.

### P-005 (new pending): Output type unions and result polymorphism
Expected as the spec library grows. fetch_url's `union<Bytes, FileWritten>`
output and similar pattern in other specs suggests output types
themselves want richer expressiveness — possibly subtype hierarchies,
discriminated unions, or generic output types parameterized by inputs.
v0.2 supports `union<>` but doesn't have idioms for richer cases.

### P-006 (new pending): Distributed and multi-host operations
Expected when LKI specs grow into multi-host or distributed
operations (querying multiple shards, fanout-then-collect patterns,
cross-region operations). v0.2 implicitly assumes single-host
execution. Distributed operations may need new capability categories
or extensions to existing ones.
