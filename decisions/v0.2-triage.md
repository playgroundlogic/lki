## Section B: Accept with modification (11 findings)

These are findings where the proposed direction is mostly right but needs adjustment.

### F-003: State mutation specifics
**Original proposal:** Formalize state_mutation with reversibility window, transaction boundaries, rollback declarations.
**Modification:** Combine with F-025's atomicity/durability proposal. State mutation specifics, durability, and atomicity are facets of the same concern: "what does it mean for this operation to succeed." v0.2 introduces a unified `mutation_semantics:` block under state_mutation capabilities covering visibility, durability, reversibility, and transactional boundaries.

### F-004: Intent as context attribute
**Original proposal:** Document that intents are not Cedar entities.
**Modification:** Accept the decision but make the rationale explicit in v0.2. The reasoning (one intent → many capability checks; making intent an entity requires synthesis or denormalization) belongs in the cedar projection chapter of v0.2, not just as a deferred-items note.

### F-008: Policy-evaluable vs resolver-enforced capabilities
**Original proposal:** Per-capability annotation `enforcement: <pdp | resolver | both>`.
**Modification:** Cleaner factoring — capabilities are *always* PDP-evaluable; resolver-only constraints are a separate category. Move resolver-only checks (regex complexity bounds, pattern length limits, encoding validation) out of `capability:` and into a new `resolver_constraints:` section. This avoids polluting the Cedar-projected capability section with non-Cedar concerns.

### F-009: Conditional template parameters
**Original proposal:** Formalize conditional mapping with `when_present:` / `when_absent:`.
**Modification:** Subsume into F-022's `cases: / when:` proposal, which is the more general mechanism. The single-discriminator case is the trivial single-when case.

### F-011: Shell quoting in interpolation
**Original proposal:** Pipe syntax `{name|shell_quoted}` for rendering modifiers.
**Modification:** Adopt the pipe syntax but be explicit that shell quoting is one of several "render modifiers." The full list for v0.2: `shell_quoted`, `url_encoded`, `json_escaped`, `yaml_quoted`, `base64`, `hex`, `regex_escaped`. The modifier set is extensible per v0.x version; new modifiers are non-breaking additions.

### F-012: cd && template pattern
**Original proposal:** Add `working_directory:` to implementation section.
**Modification:** Accept, but specify that working_directory implementation must use tool-native facilities (`git -C`, `tar -C`, `make -C`, `cwd=` on spawn) rather than shell `cd`. The format spec should explicitly forbid `cd` in templates.

### F-014: Hook execution model
**Original proposal:** Add `side_effects:` section.
**Modification:** Reframe as `internal_behaviors:` — broader than just side effects. Hooks are one example; others include retries, cache lookups, parallel sub-invocations. v0.2 introduces this section with documented examples; full formalization tracked for v0.3.

### F-019: Resource bounds as first-class
**Original proposal:** Separate `resource_bounds:` section from `inputs:`.
**Modification:** Cleaner placement — resource bounds stay in `signature.inputs:` (they're still user-facing parameters) but get an `is_resource_bound: true` annotation. This preserves the discoverability of parameters in one place while letting Cedar policies filter for them ("permit fetch_url where context.is_resource_bound parameter `timeout` is <= 30s"). Two-section approach risks parameters being defined twice or in the wrong place.

### F-022: Multi-discriminator parameter mapping
**Original proposal:** `cases:` with `when:` clauses for multi-parameter discrimination.
**Modification:** Adopt the structure but add a `default:` clause for the fall-through case. Real templates often have one "common" case and a few variations; making everyone enumerate every combination is awkward. Plus a `merge:` mode for cases where the chosen case adds to a common template rather than replacing it.

### F-023: Conditional capability requirements
**Original proposal:** Formal expression grammar for `required_if:`.
**Modification:** Use the same grammar as F-022's `when:` clauses. One expression grammar for both conditional capabilities and conditional template cases. Keeps the surface small.

### F-027: Multiple capability instances per intent
**Original proposal:** `purpose:` enum (user_facing | implementation_detail).
**Modification:** Adopt, but add a third value: `policy_required:` for cases where the policy itself effectively *requires* a capability that's not directly visible to the user (e.g., audit-log write for compliance). Audit logs and policies can filter on this distinction; tooling can validate that `implementation_detail` capabilities don't expose new attack surface beyond `user_facing` ones.

## Section C: Needs discussion (5 findings)

These are findings where the right answer isn't obvious and we should think before committing.

### F-016: Pagination/streaming for large result sets
**The tension:** Three options proposed (`bounded`, `paginated`, `streamed`), each with different Cedar consequences:
- `bounded`: simple, current v0.1 behavior, but truncates real data
- `paginated`: continuation tokens, each page a separate PDP query, sound but verbose
- `streamed`: resolver returns iterator, but Cedar's request/response model fights this

**Discussion needed:** Should v0.2 support all three or commit to one? My instinct says all three because real intents legitimately fall into different categories — `list_directory` wants paginated, `query_database` wants streamed for large result sets, `read_file` wants bounded. But supporting three modes is substantial format expansion.

**Recommendation:** Support all three. The cost is real, the alternative (forcing every intent into one mode) is worse. Each mode has clear semantics:
- `bounded`: max_size in resource_bounds; results returned in full or fail
- `paginated`: continuation_token in output; subsequent calls reuse it; each call is a separate PDP query, but request_context carries `continuation_of: <prior_query_id>` for audit
- `streamed`: resolver returns handle; iterator pulls items; PDP evaluates *once* at handle creation, audit logs each item retrieved

### F-017: Encoding handling for content reads
**The tension:** F-017 proposes either parameter-determined-output-type or separate intents per type. F-018 proposes richer compound types in catalog.

**Discussion needed:** These are related but separable. Encoding-determined-return-type is a *parametric polymorphism* question; compound output types is an *output type complexity* question. They could be resolved independently.

**Recommendation:** Separate intents per return type (`read_file` returns string with encoding parameter; `read_file_bytes` returns bytes). Reasoning:
1. Tool-SI training data is cleaner with monomorphic signatures
2. Cedar policy reasoning is cleaner ("read_file_bytes requires elevated label")
3. The proliferation of intents is bounded (text vs bytes; not many real cases)
4. Output types in compound declarations stay simple (one type per intent)

The cost is having to maintain "sister" intents (read_file / read_file_bytes), but the cost is in spec authorship, not in policy or Tool-SI training, and the latter two matter more.

### F-018: Compound output types in catalog
**The tension:** Inline pseudocode (current v0.1) vs separate types registry vs embedded type declarations in each spec.

**Discussion needed:** Where do compound types live? Three options:
1. **Inline in spec** — declared where used; each spec self-contained; but duplication across specs
2. **Per-spec `types:` section** — declared at top of spec; reusable within spec; but duplication across specs
3. **Shared registry** — declared in `registries/types.yaml`; referenced by name from specs; single source of truth

**Recommendation:** Hybrid. Shared registry for types that appear in 2+ specs (DirectoryEntry, AuditLogEntry, Match, etc.). Per-spec `types:` section for types unique to one spec. Inline pseudocode forbidden in v0.2. This matches how protobuf/openapi handle shared types and aligns with how F-031 handles entity type sharing.

The hard part: when does a type get promoted from per-spec to shared? Probably: any type referenced by ≥2 specs migrates to the registry. cedar-from-lki tooling can detect this and warn.

### F-024: Resolver pipeline
**The tension:** Not really a tension; the question is whether to make this explicit in the format spec or leave it as an implementation note.

**Discussion needed:** Should v0.2 document the resolver pipeline as part of the format, or as a separate document?

**Recommendation:** Document as part of the format. The pipeline order affects spec authoring (encoding happens before template substitution, so templates can't do encoding). Implementers writing resolvers also need it canonical. v0.2 includes a "Resolver execution model" chapter.

### F-025: Atomicity/durability semantics
**The tension:** The deepest finding. F-003 (state mutation specifics), F-025 (atomicity), and parts of F-027 (capability descriptions) all touch this. What's the actual mental model?

**Discussion needed:** Three nested questions:
1. **Visibility:** When does the change become observable to other readers? (immediately on write, on rename, on transaction commit, after sync)
2. **Durability:** What survives what kind of failure? (process crash, kernel panic, power loss, disk failure)
3. **Reversibility:** Can the operation be undone, and within what window?

These three are mostly independent. A write can be visible-immediately but non-durable (write to memory-mapped file before fsync). A commit can be durable but reversible (git reset). A network state change can be reversible-via-compensating-action (DELETE for the matching POST) but not in a transactional sense.

**Recommendation:** v0.2's state_mutation capability category gets all three as separate attributes:
```
state_mutation:
  - operation: write
    resource: "filesystem:{path}"
    visibility: immediate | rename_atomic | transactional
    durability: in_memory | committed | fsynced | fsynced_with_dir
    reversibility: irreversible | reversible_locally | reversible_until_remote_sync | undo_via_compensating_action
```

Cedar policies can require specific levels for specific resources ("payroll data writes require durability >= fsynced AND reversibility == transactional"). This is verbose but the verbosity is the value — vague atomicity claims are the bug v0.1 has.

## Section D: Defer to v0.3 (3 findings + pending)

These are real findings but not v0.2-blocking.

### F-020: Implementation tool fallback semantics
**Why defer:** The pattern only matters when an LKI deployment has heterogeneous environments where preferred tools may not be present. For Phase 0 (lki.dev launch with common-CLI specs), assume modern environments. Address in v0.3 when we have deployment evidence of where fallback matters.

### F-026: Permission grants as capability declarations
**Why defer:** Real but specialized. Currently only affects write_file's permissions parameter. v0.2 includes write_file with permissions as a regular parameter; v0.3 promotes to first-class access_grant capability when we have a second intent that needs it (probably set_file_permissions or chmod equivalent).

### Pending findings P-002, P-003, P-004
**Why defer:** These are anticipated but unobserved. Better to wait until we draft `clone_repository` and `run_container` to see what actually surfaces vs what was guessed. Pending findings stay pending in FINDINGS_v0.2.md.

## Cross-cutting decisions implied by the triage

### Decision 1: Format spec organization

v0.2 expands substantially. Suggested chapter structure:

1. **Purpose and scope** — what LKI is, what's required vs optional (Tool-SI etc.)
2. **Versioning** — three layers (format, intent, tool), with new fourth layer for entity type registry versions
3. **Spec file structure** — required and optional sections
4. **Type catalog** — including structural vs observed attributes (F-030); split into primitive/structural/compound types
5. **Interpolation grammar** — with render modifiers (F-011) and conditional cases (F-022)
6. **Capability section** — with new sub-features (resource_bounds annotation, recursion_safety, mutation_semantics)
7. **Resolver constraints section** — new, separate from capability (F-008)
8. **Internal behaviors section** — new, for hooks etc. (F-014)
9. **Implementation section** — with cases (F-022), version ranges (F-015), working_directory (F-012)
10. **Examples section** — unchanged from v0.1
11. **Anti-patterns section** — unchanged from v0.1
12. **Cedar projection rules** — naming convention (F-032), expansion rules (F-029), intent-not-entity (F-004), entity-for-capability-target (F-028)
13. **Audit log structure** — formalized (F-007)
14. **Resolver execution model** — pipeline order (F-024)
15. **Migration from v0.1** — separate document

### Decision 2: Repo organization

The shared entity type and Tool registries need a home:

```
lki/
  format/
    v0.1/
      LKI_FORMAT.md           # frozen
      FINDINGS.md             # frozen
    v0.2/
      LKI_FORMAT.md           # the v0.2 spec
      MIGRATION_FROM_v0.1.md  # how to migrate
  types/
    v0.1/
      Principal.lki.yaml
      FilePath.lki.yaml
      Host.lki.yaml
      ...
  registries/
    tools/
      v0.1/
        tools.lki.yaml        # known Tool entities
  specs/
    cli/
      fetch_url/
        v1.0.0.yaml           # frozen v0.1 format
        v2.0.0.yaml           # v0.2 migration (when ready)
      ...
  projections/
    fetch_url/
      v1.0.0/
        schema.cedarschema
        policies.cedar
        pdp-query.yaml
```

This puts everything versioned cleanly and lets format/types/registries evolve independently.

### Decision 3: Migration scope for existing specs

The seven v0.1 specs all need migration to v0.2:
- All require new structural changes (resource_bounds annotation, resolver_constraints split, mutation_semantics for write_file/commit_changes)
- Intent versions bump to 2.0.0 across the board (signature changes from F-019 and F-008)
- Cedar projections bump correspondingly

Migration is mechanical but tedious. Probably one working session after v0.2 format spec is drafted.
