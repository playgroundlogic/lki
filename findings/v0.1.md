# LKI Format Findings — accumulating toward v0.2

**Source format version:** 0.1
**Status:** open; will close when v0.2 ships
**Purpose:** record format-level issues surfaced through projection
exercises and spec drafting, so v0.2 has a concrete agenda rather than an
open-ended redesign.

## Source attribution

Findings are tagged with where they came from:

- `[scoping]` — known limitation from original v0.1 design
- `[projection]` — surfaced by Cedar projection of an LKI spec
- `[drafting]` — surfaced by writing a spec against v0.1

Each finding describes the issue, why it matters, and a proposed direction
for v0.2. Proposals are not commitments; v0.2 may handle differently.

---

## From original v0.1 scoping

### F-001: Output schemas use inline pseudocode
**Source:** `[scoping]`
**Description:** v0.1 specs declare complex return types using `schema: |`
with inline pseudocode strings. Consumers can't programmatically parse
these.
**Why it matters:** Tool-SI training, audit log structure, and resolver
return types all need machine-readable output schemas. Pseudocode prose
defers the problem.
**Proposed direction:** v0.2 formalizes output schemas using the same
type catalog used for inputs, including compound types like
`Match { path: FilePath; line_number: int; line: string }`.

### F-002: Endless integration not declared
**Source:** `[scoping]`
**Description:** No way to declare which fields should trigger
gap-detection queries when missing or under-specified.
**Why it matters:** When endless is integrated, vague intent should
become clarifying questions rather than policy denials at runtime.
**Proposed direction:** v0.2 adds optional `gap_detection:` annotations
per parameter and per constraint, declaring what endless should ask
about when the field is missing or unbounded.

### F-003: State mutation semantics underspecified
**Source:** `[scoping]`
**Description:** The `state_mutation` capability category exists in v0.1
but the semantics of `reversible`, the relationship to transactions, and
rollback declarations are underspecified.
**Why it matters:** Several real intents (commit, send_email, db_insert)
are state-mutating in ways that policy and audit should reason about
distinctly from filesystem write.
**Proposed direction:** v0.2 formalizes state_mutation with explicit
fields for reversibility window, transaction boundaries, and rollback
declarations.

## From cedar projection of fetch_url@1.0.0

### F-004: Intent as context attribute, not Cedar entity
**Source:** `[projection]`
**Description:** Cedar projection puts `intent_name` and `intent_version`
in request context rather than making the intent a Cedar entity. The
reasoning is that one intent invocation produces multiple capability
checks, and making the intent an entity would require either synthesizing
a transient entity per invocation or denormalizing each capability.
**Why it matters:** Document the decision explicitly so it's preserved
across implementations and isn't re-debated per projection exercise.
**Proposed direction:** v0.2 formally documents that intents are not
Cedar entities; intent identity flows through context attributes.

### F-005: Derived attributes on types
**Source:** `[projection]`
**Description:** Cedar policies rely heavily on derived attributes
(`Host.is_metadata`, `FilePath.is_system_path`, `FilePath.is_within_working_dir`)
that aren't declared in LKI parameter definitions. They're guaranteed by
the type during canonicalization.
**Why it matters:** Different implementations might derive different
attributes, breaking policy portability across LKI runtimes.
**Proposed direction:** v0.2's type catalog declares which derived
attributes each type guarantees. The list becomes part of the type
contract.

### F-006: Intent-level atomicity not specified
**Source:** `[projection]`
**Description:** Cedar evaluates each request independently, but intents
are atomic across their capability set — partial-allow is never a valid
outcome. v0.1 doesn't state this explicitly.
**Why it matters:** A naive implementation might allow a fetch_url that
permits the network but not the filesystem write to silently drop the
write. This is wrong: the intent must fail atomically.
**Proposed direction:** v0.2 formalizes "an intent that requires N
capabilities requires all N PDP decisions to be Allow; any Deny fails
the intent atomically and returns a structured error identifying which
capability was denied."

### F-007: Audit log structure
**Source:** `[projection]`
**Description:** The audit log shape falls out naturally from projection
and is consistent enough across intents to be declarable at the format
level.
**Why it matters:** All LKI implementations should produce compatible
audit streams. Audit log compatibility is what enables queryable
historical analysis, policy iteration, and Tool-SI training data
refinement.
**Proposed direction:** v0.2 formalizes the audit entry structure:
intent, intent_version, principal, resolved capabilities, PDP decisions
with determining policies, execution outcome, timing. Content-addressable
entries with append-only semantics.

### F-008: Policy-evaluable vs resolver-enforced capabilities
**Source:** `[projection]`
**Description:** Some capabilities (`compute.bounded_regex`) can't
project to Cedar because Cedar doesn't reason about regex complexity.
These are resolver-side checks that reject before PDP is consulted, not
policy-side checks.
**Why it matters:** Both belong in the LKI capability section but have
different downstream consumers. Without explicit distinction, generator
tools have to guess which capabilities are PDP-bound.
**Proposed direction:** v0.2 annotates each capability with explicit
`enforcement: <pdp | resolver | both>` declaration. Resolver-only
capabilities are documented in the spec but not projected to Cedar.

## From drafting commit_changes@1.0.0

### F-009: Conditional template parameters awkward in v0.1 grammar
**Source:** `[drafting]`
**Description:** `commit_changes` needs a `stage_command` template part
that differs based on whether `paths` parameter is provided. The v0.1
`parameter_mapping` syntax had to use a `paths_provided` / `paths_omitted`
discriminator that isn't formally part of the grammar.
**Why it matters:** Many real intents will have conditional template
fragments. v0.1's workaround pattern is inconsistent across specs.
**Proposed direction:** v0.2 formalizes conditional mapping, probably as
explicit `when_present:` / `when_absent:` keys in parameter_mapping for
optional parameters.

### F-010: Composite/derived parameters need formal handling
**Source:** `[drafting]`
**Description:** `commit_changes` needed an `author_combined` value
formed from `author_name` and `author_email` formatted as
`"Name <email>"`. The v0.1 grammar can't declare derived parameters.
**Why it matters:** Composition of parameter values is a common pattern
(URL building, formatted display, escape sequences).
**Proposed direction:** v0.2 introduces type methods that take other
parameters as arguments, or explicit `derived_parameters:` declarations
with formula expressions.

### F-011: Shell quoting not declarable
**Source:** `[drafting]`
**Description:** `commit_changes` informally used `{message[shell_quoted]}`
notation for shell-quoted message rendering. Not in the v0.1 grammar.
**Why it matters:** Shell quoting is non-negotiable for any intent
whose template invokes a shell. Without explicit declaration, every
spec invents its own quoting convention.
**Proposed direction:** v0.2 adds rendering modifiers to interpolation
grammar via pipe syntax: `{message|shell_quoted}`. Standard modifiers
include `shell_quoted`, `url_encoded`, `json_escaped`, `base64`, etc.

### F-012: `cd && ...` pattern in templates is wrong
**Source:** `[drafting]`
**Description:** The commit_changes template used `cd {repository_path}
&& git ...` for readability, but production resolvers should use
`git -C {repository_path}` to avoid working-directory side effects.
**Why it matters:** The template should not encode shell-side-effect
patterns; the resolver's execution context should be declared
explicitly.
**Proposed direction:** v0.2 adds `working_directory:` to the
implementation section, declaring the directory context for tool
invocation. Templates assume execution in that directory; resolvers
use tool-native options (`git -C`, `tar -C`, `make -C`, etc.) or
spawn the tool with appropriate `cwd` rather than relying on cd.

### F-013: Constraint placement inconsistent
**Source:** `[drafting]`
**Description:** Some constraints live in the parameter definition
(`constraint: "must be a valid email"`), some in the signature
`constraint:` blocks, some in the top-level `constraints:` section.
v0.1 was ambiguous about which goes where.
**Why it matters:** Tooling that validates specs against the format
needs unambiguous rules about where to find each kind of constraint.
**Proposed direction:** v0.2 specifies:
  - Parameter-level constraints (per-parameter validity rules) → inside
    parameter definition
  - Cross-parameter constraints (rules involving multiple parameters) →
    top-level `constraints:`
  - Behavioral constraints (rules about the intent's effects beyond
    parameter validation) → in `behavior` prose; not machine-parseable

### F-014: Hook execution model not articulated
**Source:** `[drafting]`
**Description:** Git hooks run during commit and can fail the commit.
v0.1 commit_changes doesn't explicitly model "hooks run" as either a
capability or a constraint. Hooks are arbitrary code execution that
happens during the LKI operation.
**Why it matters:** From a policy perspective, hooks are something
agents may want to permit or forbid. Without a model, hook execution
is implicit and policy can't reason about it.
**Proposed direction:** v0.2 adds a `side_effects:` section to capture
intent-internal behaviors that aren't direct capability uses but matter
for policy. Hook execution declared as a side_effect; policy can
restrict whether hooks are permitted to run, which hook types, etc.

### F-015: Tested-against versions need range syntax
**Source:** `[drafting]`
**Description:** commit_changes lists three exact git versions in
`tested_against`. Most specs will want to declare "tested against
git 2.x" rather than enumerating patch releases.
**Why it matters:** Exact version enumeration doesn't scale; ranges or
patterns are what's actually useful for compatibility declarations.
**Proposed direction:** v0.2 supports semver ranges and patterns in
`tested_against`: `["git 2.30 - 2.42", "git 2.x"]`. Same syntax as
semver-compatible ecosystems.

## From drafting read_file@1.0.0 and list_directory@1.0.0

### F-016: Pagination/streaming for large result sets
**Source:** `[drafting]` (promoted from P-001)
**Description:** `list_directory` for any non-trivial directory immediately
hits the question of result-set size. v1.0.0 worked around it with a
`max_entries` resource bound and a `truncated: bool` output flag, but
truncation doesn't give the caller a way to retrieve the rest.
**Why it matters:** Many real intents will produce result sets larger
than fits in memory or context: `list_directory` on populated trees,
`grep_files` on large codebases, audit log queries. Without a streaming
or pagination model, every such intent has to invent its own
truncation/continuation pattern.
**Proposed direction:** v0.2 adds a `streaming:` declaration in the
output section: either `bounded` (current v0.1 behavior with max_entries),
`paginated` (returns continuation token; next call resumes), or `streamed`
(resolver returns a handle the caller can iterate). The pagination model
needs careful design — Cedar policy still applies per-page-fetch but
must remain consistent within a single logical query.

### F-017: Encoding handling for content reads
**Source:** `[drafting]`
**Description:** `read_file` exposes an `encoding` parameter with values
including `utf_8`, `utf_16`, `latin_1`, `ascii`, `bytes`. The return
type is `union<string, bytes>` depending on the encoding choice. This
works but feels awkward — the parameter changes the return type, which
is unusual for typed parameter signatures.
**Why it matters:** Any intent that returns content from external
sources (read_file, fetch_url, query_database with text/binary returns)
faces this design question. Some return raw bytes, some return decoded
strings; the choice has historically been ad-hoc per tool.
**Proposed direction:** v0.2 may handle this with overloaded signatures
(`read_file` vs `read_file_bytes` as separate intents at the format level)
or with explicit `parameter_determines_output_type:` annotations that
make the dependence formal. The current union-output approach works but
should be documented as a v0.1 limitation if v0.2 unifies it.

### F-018: Compound output types need first-class catalog support
**Source:** `[drafting]`
**Description:** `list_directory` returns `DirectoryListing` containing
`list<DirectoryEntry>`, where `DirectoryEntry` itself is a compound type
with fields. Declared in inline pseudocode per F-001. The v0.1 type
catalog doesn't have a way to declare these compound types, so each
spec invents its own.
**Why it matters:** Audit log entries, query result rows, container
inspection results, git log entries — many intents produce structured
compound outputs. Declared per-spec means: no shared types between
specs that should share them, no Cedar context attributes for output
shape, no Tool-SI training data type-aware.
**Proposed direction:** v0.2 introduces a `types:` registry at the spec
file level (or a separate file referenced by name) where compound types
are declared once and referenced by name. Cross-spec sharing happens by
referencing the same type name from a shared registry.

### F-019: Resource bounds as first-class declarations
**Source:** `[drafting]`
**Description:** `read_file` declares `max_size: Bytes` and `list_directory`
declares `max_entries: int` as input parameters that bound the operation's
resource use. These are conceptually different from "user-provided values":
they're caps on the operation's appetite for resources, with policy
implications and consistent semantics across many intents.
**Why it matters:** Resource bounds appear in many intents (max_response_size
on fetch_url, max_pattern_complexity on grep_files, max_entries on list_directory,
timeout on multiple). They're often the safety-floor parameters that policy
wants to constrain. Mixing them with regular parameters obscures their role.
**Proposed direction:** v0.2 adds a `resource_bounds:` section separate
from `inputs:` for parameters that cap resource use. Cedar policies can
reason specifically about resource_bounds (e.g., "this principal can
request max_size up to 100MB; larger requires elevated label").

### F-020: Implementation tool fallback semantics
**Source:** `[drafting]`
**Description:** `list_directory` declares both `find` (preferred) and `ls`
(fallback) implementations with different capability profiles — ls
needs more resolver-side work to produce the structured output.
v0.1 has no formal "fallback" semantics; the resolver picks the first
available `preferred: true` implementation, or the first in declaration
order.
**Why it matters:** When the preferred implementation isn't available
(unusual environments, BusyBox systems, restricted containers), the
resolver should have explicit guidance about when to fall back and
whether the fallback is policy-equivalent.
**Proposed direction:** v0.2 formalizes fallback selection: each
implementation declares a `priority:` integer, the resolver tries highest
first, falls through on tool-absence. Critically, all implementations
must produce policy-equivalent outputs — if a fallback has different
capability requirements, that's a separate intent, not a fallback.

### F-021: Visited-set / cycle prevention as resolver responsibility
**Source:** `[drafting]`
**Description:** `list_directory` recursive mode with symlink following
requires cycle prevention. v0.1 spec notes "visited-set tracking at the
resolver layer" but doesn't formalize how the resolver knows to do this.
**Why it matters:** Any recursive intent (recursive grep, recursive
directory listing, recursive dependency walk) faces cycle risk. Without
format-level declaration, each resolver implements cycle prevention
ad-hoc.
**Proposed direction:** v0.2 capability category adds `recursion_safety:`
declarations: `cycle_prevention: required | not_applicable`,
`max_depth: int | unbounded`, `visited_set: by_canonical_path | by_inode`.
Resolvers consult these to implement consistent behavior.

## From drafting write_file@1.0.0

### F-022: Multi-discriminator parameter mapping
**Source:** `[drafting]`
**Description:** `write_file` requires templates that vary across the
combination of `atomic` × `mode` (atomic × {replace, append, create_new})
producing six cases. v0.1's parameter_mapping syntax handles one
discriminator cleanly but multiple discriminators produce a flat namespace
(`atomic_replace`, `atomic_append`, `atomic_create_new`, `nonatomic_replace`,
etc.) that doesn't scale.
**Why it matters:** Real intents have multi-dimensional implementation
choice. Any intent that has more than one mode-like parameter will face
this. write_file is the first; container operations, network operations
with auth modes, and database operations with isolation levels will all
hit it harder.
**Proposed direction:** v0.2 introduces a `cases:` declaration with
explicit when-clauses for multi-parameter discrimination:
```
template:
  cases:
    - when: { atomic: true, mode: replace }
      command: |
        tmpfile=$(...) && printf ... && mv ...
    - when: { atomic: true, mode: append }
      command: ...
```
Cleaner than flat namespace and naturally extends to N discriminators.

### F-023: Conditional capability requirements
**Source:** `[drafting]`
**Description:** write_file declares a capability `filesystem.write` on
`{path.parent}` that's only required when `atomic` is true OR
`create_parents` is true. v0.1 uses `required_if: "atomic is true or
create_parents is true"` as a string expression, which is not part of
the v0.1 grammar.
**Why it matters:** Conditional capabilities are common in real intents.
fetch_url's filesystem write is conditional on output being provided.
Database queries' write capability is conditional on the query being
mutating. Without formal grammar, each spec invents its own conditional
expression syntax.
**Proposed direction:** v0.2 formalizes a small expression grammar for
`required_if:` conditions — boolean expressions over input parameters
and their presence. Same grammar should be used for the conditional
interpolation noted in F-009.

### F-024: Encoding handled at resolver, invisible to template
**Source:** `[drafting]`
**Description:** write_file's `encoding` parameter is applied at the
resolver layer before the content is substituted into the template.
The template uses `{content[shell_quoted]}` but the content is already
in its encoded byte form by then. The template sees the post-encoding
bytes, not the original string.
**Why it matters:** This means encoding is a *resolver responsibility*
not a *template responsibility*, and the template/parameter-mapping
grammar doesn't need encoding-aware rendering. Worth documenting
explicitly so spec authors don't try to do encoding in templates.
**Proposed direction:** v0.2 formalizes the "resolver pipeline":
parameter validation → encoding transformations → conditional resolution
→ template substitution → shell quoting → execution. Spec authors think
about parameters; the resolver handles the pipeline. Template syntax
only needs to address what the substitution layer can see.

### F-025: Write atomicity semantics need explicit declaration
**Source:** `[drafting]`
**Description:** write_file declares `atomic: bool` as a parameter
controlling whether writes go through temp-file-and-rename. But the
*meaning* of "atomic" depends on what's being protected against —
concurrent readers, process crashes, kernel crashes, power failures.
v0.1 hand-waves; the actual guarantees depend on filesystem (ext4 vs
NFS vs Object FS), tools (mv across filesystems vs same filesystem),
and write modes (append's O_APPEND atomicity vs replace's rename).
**Why it matters:** Cedar policy should be able to reason about
durability and atomicity properties as resource attributes. "This
agent can write to /tmp without atomicity guarantees; production
data writes require atomicity + fsync." Without formal semantics,
this is impossible to express.
**Proposed direction:** v0.2 adds explicit durability/atomicity
declarations to the state_mutation capability category:
```
state_mutation:
  - operation: write
    resource: "filesystem:{path}"
    visibility_atomicity: rename_atomic | append_atomic | nonatomic
    durability: in_memory | fsynced | fsynced_with_dir
```
Cedar policies can require specific durability levels for specific
resources.

### F-026: Permission grants as capability declarations
**Source:** `[drafting]`
**Description:** write_file's `permissions` parameter sets file mode
bits, which is a capability transfer (the file's future readers/writers/
executors are determined by these bits). v0.1 treats it as a regular
parameter, but it's conceptually closer to a capability *output* than
an *input*.
**Why it matters:** Cedar policy should reason about what permissions
the agent is *granting* to the file, not just whether the agent can
write. "Agent may write but may not grant execute permission on the
resulting file" is a real policy. write_file v0.1 mixes this into the
parameter validation rather than making it a first-class capability.
**Proposed direction:** v0.2's capability section adds a new category
`access_grant:` for cases where an operation creates or modifies the
access rights of a resource. write_file would declare:
```
access_grant:
  - operation: set_mode
    resource: "filesystem:{path}"
    mode: "{permissions}"
```
Cedar can then express "no grants of world-write," "no grants of execute
in workspace paths," etc.

### F-027: Multiple capability instances per intent
**Source:** `[drafting]`
**Description:** write_file has *two* `filesystem.write` capability
declarations (one for the target path, one for the parent path under
specific conditions). v0.1 syntactically supports this (list of
capabilities) but the runtime PDP semantics aren't specified. Both
checks must pass for the intent to proceed; F-006 (atomicity) covers
the all-or-nothing semantic, but the question of "are these two
permits or one combined permit" is open.
**Why it matters:** Audit log structure (F-007) needs to distinguish
between "agent wrote to target file" and "agent's atomic-write
infrastructure needed parent dir access" — they're semantically
different even though both are filesystem.write.
**Proposed direction:** v0.2 supports an optional `description:` field
per capability declaration (already used informally in v0.1) and adds
a `purpose:` enum that distinguishes user-facing capability use from
implementation-detail capability use. Audit logs include both but can
filter on purpose. Cedar policies can write rules that ignore
implementation-detail capability use ("if the user-facing write is
permitted, the implementation-detail temp-file write is too" — saves
operators from writing redundant policy rules).

## From cedar projection of commit_changes@1.0.0

### F-028: Cedar requires a resource even when none is natural
**Source:** `[projection]`
**Description:** `Process::Spawn` has no natural resource entity — the
intent spawns a process, but Cedar's request shape requires a resource
attribute. The projection resolved this by introducing `Tool` as an
entity type where each spawnable tool is a known entity (`Tool::"git"`,
`Tool::"curl"`, etc.). Other options considered (Principal-as-resource,
Process-as-transient-entity, singleton System entity) all had drawbacks.
**Why it matters:** Several future capability categories will face the
same question. Compute capabilities (CPU, memory, GPU bounds) don't have
natural resources. Time/timeout capabilities don't either. Without
a consistent answer, the schema accumulates ad-hoc resource types per
capability category.
**Proposed direction:** v0.2 declares the answer formally: when a
capability has no natural resource entity, an entity type representing
the *capability target* is introduced (Tool for process.spawn, ComputeBudget
for compute.bounded, TimeBudget for timeout). This is verbose but matches
Cedar's design and makes policies read naturally ("permit spawn of
Tool::git" rather than "permit spawn with tool='git' context attribute").

### F-029: One capability declaration produces N PDP queries
**Source:** `[projection]`
**Description:** commit_changes' first filesystem.read capability lists
two paths: `["{repository_path}/.git", "{repository_path}"]`. The
projection produces *two separate PDP queries*, one per resolved path.
The relationship between "capability declaration" and "PDP query count"
is not 1:1 in v0.1.
**Why it matters:** Operators reading the LKI spec see one capability
declaration; the audit log shows multiple decisions; the relationship
between them isn't documented anywhere except by inference. Tool-SI
training data will also need to know that multiple PDP queries are
expected per capability declaration.
**Proposed direction:** v0.2 documents the expansion rule explicitly: a
capability declaration with a list-valued resource field expands into
one PDP query per resource. If authors want one combined query, they
should declare separate capabilities. Make the implicit explicit so
the audit log structure aligns with operator mental model.

### F-030: Request-time state observation vs spec-time declaration
**Source:** `[projection]`
**Description:** `GitRepository` entity attributes like `current_branch`
and `has_uncommitted_changes` are not declared by the spec — they're
*observed* at request construction time from the actual repository state.
The spec declares which attributes exist; the resolver populates them
when the intent fires. v0.1 doesn't distinguish "attributes that exist
on every instance of this type" from "attributes that are observed per
request."
**Why it matters:** Cedar policies write rules over these observations.
Spec authors and policy authors need to understand which attributes are
durable (the type catalog) versus point-in-time (the request observation).
Policies that depend on point-in-time attributes have TOCTOU implications
that aren't obvious without the distinction.
**Proposed direction:** v0.2 splits entity type attributes into
`structural:` (declared by type, stable for entity identity) and
`observed:` (populated per-request from environment state). The
distinction shows up in the schema syntax and is documented as
load-bearing for policy authors.

### F-031: Cross-spec entity type sharing
**Source:** `[projection]`
**Description:** `Principal` and `FilePath` are entity types used by
fetch_url, grep_files, remove_files, read_file, list_directory,
write_file, AND commit_changes. The projection of any individual spec
shouldn't redeclare these; there's a shared namespace they belong to.
v0.1 doesn't say where entity types live or how cross-spec sharing
works.
**Why it matters:** When cedar-from-lki generates the unified schema,
it walks all specs and merges entity type declarations. Conflicts
between specs ("FilePath declared with different attributes in two
specs") need a resolution rule. Adding a new attribute to a shared
entity affects all specs that use it.
**Proposed direction:** v0.2 separates entity type catalog from
per-spec capability declarations. Entity types are declared in a
shared registry (`types.lki.yaml` or similar). Specs reference type
names; types are versioned independently. Adding a new attribute to
a shared type is a minor version bump on the type, not on the specs
that use it.

### F-032: Multi-spec namespace organization
**Source:** `[projection]`
**Description:** The commit_changes projection added 3 new actions
(Filesystem::Read, StateMutation::Write::Git, Process::Spawn) to the
namespace introduced by fetch_url. The naming convention
(Category::Verb::Specifier) emerged organically — `Filesystem::Write`,
`Filesystem::Read`, `StateMutation::Write::Git`, `Process::Spawn`. v0.1
doesn't formalize this naming.
**Why it matters:** As more specs add actions, naming consistency
matters. Without a convention, we'll end up with `FsWrite`, `FilesystemRead`,
`WriteFs`, etc. across different specs.
**Proposed direction:** v0.2 specifies the naming convention as part of
the cedar projection rules: `<CapabilityCategory>::<Verb>` or
`<CapabilityCategory>::<Verb>::<Specifier>`. Capability categories are
the keys in the LKI capability section (filesystem, network, process,
state_mutation, compute). Verbs are the operations (read, write, spawn,
etc.). Specifiers exist only where multiple sub-categories are needed
(StateMutation::Write::Git vs StateMutation::Write::Email).

### F-033: Tool entity registry maintenance
**Source:** `[projection]`
**Description:** The Tool entity type implies a registry of known tools
(`Tool::"git"`, `Tool::"curl"`, etc.). This registry must be maintained
alongside the LKI specs — adding a new tool to any spec requires adding
the Tool entity to the registry. Without an explicit place this lives,
it'll drift.
**Why it matters:** Operators deploying LKI need to know which tools
are declared. Cedar policies reference tools by name; typos in tool
names produce silent policy mismatches. The Tool registry is a
deployment artifact, not just a schema concern.
**Proposed direction:** v0.2 specifies that the Tool registry (and any
other "instance registries" the schema requires) live in `registries/`
adjacent to specs. cedar-from-lki validates that every Process::Spawn
capability references a Tool that exists in the registry; missing tools
produce an explicit error rather than silently policy-allowing or
policy-denying. The registry is content-addressable and versioned like
specs themselves.

## Pending — not yet observed but expected

These are anticipated findings that haven't surfaced in drafted specs
yet but are likely to during the next batch.

### P-002: Authentication context propagation
Expected from `clone_repository` and other intents that need credentials
without exposing them in parameters. SSH keys, OAuth tokens, etc.

### P-003: Process tree spawning vs single process
Expected from `run_container` where the container itself spawns processes.
The current `process.spawn` capability only covers the direct child.

### P-004: Network policy with allowlist/denylist semantics
Expected from any spec that needs to allow some hosts but not others
based on policy. v0.1 capability declarations are descriptive (what
the intent needs) rather than prescriptive (what policy permits).
