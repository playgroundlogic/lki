# Migrating LKI Specs from v0.1 to v0.2

**Companion to:** [LKI_FORMAT_v0.2.md](../format/v0.2/LKI_FORMAT.md)
**Status:** Authoritative
**Last updated:** 2026-05-19

## Overview

v0.2 incorporates 33 findings from v0.1 use, organized as 25 mechanical
integrations and 5 substantive design decisions. This guide describes
how to migrate existing v0.1 specs to v0.2.

Migration is necessary, not optional, for specs that need v0.2 tooling
or v0.2 feature integration. v0.1 specs continue to be readable by
v0.2-aware tooling but cannot use v0.2 features (resource bounds
annotation, three-axis state mutation, conditional cases, etc.).

### Migration scope

Every v0.1 spec becomes a new file at a new intent version (2.0.0).
The original v0.1 spec is preserved as historical record in the
versioned spec directory. Migration is per-file, not in-place editing.

For example:
- `specs/cli/fetch_url/v1.0.0.yaml` (v0.1, frozen, historical)
- `specs/cli/fetch_url/v2.0.0.yaml` (v0.2, the migrated version)

### What's automatable vs what needs judgment

**Automatable (mechanical migrations):**
- Format version field update
- Intent version major bump
- Constraints section restructuring
- Type reference version pinning syntax
- Anti-pattern structure (no changes needed)
- Examples structure (no changes needed)

The `lki-migrate` tool handles these mechanically. Running it produces
a v0.2 draft of each input spec; the draft requires further human
work for the substantive items.

**Requires judgment (substantive migrations):**
- Resource bounds annotation (which parameters bound resources?)
- Resolver constraints extraction (which checks are non-Cedar?)
- State mutation three-axis decomposition (what does the operation
  actually guarantee?)
- Internal behaviors documentation (what side behaviors exist?)
- Capability purpose annotation (user-facing vs implementation-detail?)
- Working directory declaration (where do `cd && ...` patterns live?)
- Conditional cases for templates (multi-discriminator patterns?)
- Tool registry references (which tools does this spec spawn?)

This guide treats each substantive item in its own section.

## Part I — Mechanical migrations

These changes are syntactic and can be applied automatically.

### M1. Format version field

Update the top-level field:

```yaml
# v0.1
lki_version: "0.1"

# v0.2
lki_version: "0.2"
```

### M2. Intent version major bump

v0.1 to v0.2 is a format-level breaking change. All migrated intents
get a new major version:

```yaml
# v0.1
spec:
  intent: fetch_url
  intent_version: "1.0.0"

# v0.2
spec:
  intent: fetch_url
  intent_version: "2.0.0"
```

This is the bump even when the spec's *semantic* contract is unchanged.
The intent version reflects format compatibility, not just behavioral
changes.

### M3. Constraints section restructuring

v0.1 had a flat `constraints:` list at the spec level. v0.2 splits
this into two subsections:

```yaml
# v0.1
constraints:
  - "url must include scheme"
  - "max response size: 100MB"
  - "If method is HEAD, output cannot be specified"

# v0.2
constraints:
  parameter_constraints:
    - parameter: url
      rule: "must include scheme (http or https)"
    - parameter: max_size
      rule: "must be <= 100MB"
  cross_parameter_constraints:
    - description: "HEAD method incompatible with output destination"
      rule: "method == HEAD implies output is null"
```

When a constraint involves only one parameter, prefer using the inline
`constraint:` field on the parameter itself (Chapter 4.1 of v0.2)
rather than `parameter_constraints`. The `parameter_constraints`
subsection exists for cases where the constraint is too long or
complex for inline expression.

When a constraint involves multiple parameters, it must go in
`cross_parameter_constraints` with both a human-readable `description`
and a machine-evaluable `rule` (using the expression grammar from
Chapter 6).

### M4. Type reference version pinning

v0.1 used bare type names (`type: FilePath`). v0.2 requires version
pinning for non-primitive types:

```yaml
# v0.1
inputs:
  - name: url
    type: URL
  - name: path
    type: FilePath

# v0.2
inputs:
  - name: url
    type: URL@1.x
  - name: path
    type: FilePath@1.x
```

Primitive types (string, int, bool, bytes) do not require version
pinning. Compound types (list<T>, map<K,V>, enum<>, union<>) embed
version pins on their type arguments where applicable.

The pinning prevents future major version bumps of shared types from
silently changing spec semantics.

### M5. Examples and anti-patterns

The structures of these sections are unchanged from v0.1. Existing
content carries over unchanged.

Minor enhancement: v0.2 supports `$ref:` references to shared
anti-pattern registries (Chapter 14.5). For migration, this is
optional — `$ref` is used only when an anti-pattern is genuinely
shared across multiple specs. For the v0.1 to v0.2 migration, leave
anti-patterns inline.

### M6. Tested-against version syntax

v0.1 typically listed exact versions:

```yaml
# v0.1
tested_against: ["git 2.34.1", "git 2.39.0", "git 2.42.0"]
```

v0.2 supports the same exact-version syntax but also ranges:

```yaml
# v0.2 (equivalent, more concise)
tested_against: ["git 2.30 - 2.42"]
```

This is optional. Existing exact-version lists are still valid in v0.2.

## Part II — Substantive migrations

These changes require judgment about the intent's semantics.

### S1. Resource bounds annotation (was F-019)

Parameters that cap resource use need `is_resource_bound: true`. The
question is: which parameters are bounds?

A parameter is a resource bound if:
1. It limits the size, duration, count, or cost of the operation
2. Its purpose is to prevent runaway resource consumption
3. Cedar policies might want to write rules over it (e.g., "this
   principal can request timeout up to 30s; larger requires elevated
   label")

Examples across v0.1 specs:

| Spec | Resource-bound parameters |
|------|---------------------------|
| fetch_url | `timeout`, `max_size` (if it had been declared) |
| grep_files | `max_count`, `context_lines` |
| read_file | `max_size` |
| list_directory | `max_entries` |
| write_file | none typically (size comes from content, not a bound parameter) |
| commit_changes | none typically |
| remove_files | none typically |

Migration:

```yaml
# v0.1
inputs:
  - name: timeout
    type: Duration
    default: 30s
    constraint: "must be <= 300s"

# v0.2
inputs:
  - name: timeout
    type: Duration@1.x
    default: 30s
    constraint: "must be <= 300s"
    is_resource_bound: true
```

The `is_resource_bound: true` annotation causes the parameter to flow
into Cedar context as `context.<parameter_name>_bound` for every
capability the intent declares (Chapter 9.3.2).

### S2. Resolver constraints extraction (was F-008)

v0.1 mixed Cedar-enforceable capabilities with resolver-side checks.
v0.2 separates them. Resolver-side checks move out of `capability`
into the new `resolver_constraints` section.

A check is resolver-side (not Cedar-side) if:
1. Cedar's semantics can't reason about it
2. It involves syntactic parsing or pattern analysis
3. It's enforced before any PDP query

Common resolver-side checks:
- Regex complexity bounds (ReDoS prevention)
- Encoding validation
- Shell metacharacter rejection in path/string parameters
- Format-level validation (URL parseable, JSON valid, etc.)

Migration example for grep_files:

```yaml
# v0.1
capability:
  filesystem:
    - operation: read
      paths: "{paths}"
      recursive: "{recursive}"
      follow_symlinks: false
  compute:
    - bounded_regex: true
      max_pattern_complexity: "no_nested_quantifiers, no_recursion, no_lookahead_when_perl"

# v0.2
capability:
  filesystem:
    - operation: read
      paths: "{paths}"
      recursive: "{recursive}"
      follow_symlinks: false
      purpose: user_facing
      recursion_safety:
        cycle_prevention: required
        max_depth: 100
        visited_set: by_canonical_path

resolver_constraints:
  pattern_complexity:
    - parameter: pattern
      rule: "no nested quantifiers when grammar is perl"
      reason: "ReDoS prevention"
    - parameter: pattern
      rule: "pattern length <= 1024"
      reason: "Bounded parsing time"
```

The `compute.bounded_regex` declaration from v0.1 wasn't actually a
Cedar-projectable capability — Cedar can't reason about regex
complexity. It belongs in resolver_constraints.

### S3. State mutation three-axis decomposition (was F-003, F-025)

v0.1's state_mutation declarations had vague semantics. v0.2 requires
three independent attributes:

- **visibility**: when does the change become observable?
  (`immediate` | `rename_atomic` | `transactional`)
- **durability**: what survives what kind of failure?
  (`in_memory` | `committed` | `fsynced` | `fsynced_with_dir`)
- **reversibility**: can the operation be undone?
  (`irreversible` | `reversible_locally` | `reversible_until_remote_sync` | `undo_via_compensating_action`)

This is the most substantive migration. Spec authors must think about
what the operation actually guarantees.

Migration examples:

**commit_changes:**

```yaml
# v0.1
state_mutation:
  - operation: write
    resource: "git_repository:{repository_path.canonical}"
    reversible: true

# v0.2
state_mutation:
  - operation: write
    resource_type: GitRepository@1.x
    resource_identifier: "{repository_path.canonical}"
    visibility: immediate
    durability: committed
    reversibility: reversible_until_remote_sync
    purpose: user_facing
```

Reasoning:
- **visibility: immediate** — once `git commit` returns, other readers
  see the new commit immediately via `git log`, `git status`, etc.
- **durability: committed** — the commit is in `.git/objects/` and
  refs are updated, but git doesn't `fsync` by default. Power failure
  could theoretically lose recent commits.
- **reversibility: reversible_until_remote_sync** — `git reset` undoes
  the commit until it's pushed to a remote. After push, history rewrite
  is needed (effectively irreversible from this intent's perspective).

**write_file (depends on atomic parameter):**

For atomic=true, fsync=false:
```yaml
state_mutation:
  - operation: write
    resource_type: FilePath@1.x
    resource_identifier: "{path.canonical}"
    visibility: rename_atomic
    durability: committed
    reversibility: irreversible
    purpose: user_facing
```

For atomic=true, fsync=true:
```yaml
state_mutation:
  - operation: write
    resource_type: FilePath@1.x
    resource_identifier: "{path.canonical}"
    visibility: rename_atomic
    durability: fsynced
    reversibility: irreversible
    purpose: user_facing
```

For atomic=false:
```yaml
state_mutation:
  - operation: write
    resource_type: FilePath@1.x
    resource_identifier: "{path.canonical}"
    visibility: immediate
    durability: in_memory
    reversibility: irreversible
    purpose: user_facing
```

Note that since write_file's three-axis values depend on parameters,
the spec needs either conditional cases in the state_mutation
declaration (using `required_if:` and per-case declarations) or
multiple state_mutation entries with `required_if:` distinguishing
them.

This is one place where the multi-discriminator pattern (M9 below)
intersects state mutation migration.

### S4. Internal behaviors documentation (was F-014)

v0.1 specs didn't explicitly document side behaviors (hooks, retries,
cache lookups, sub-process spawning). v0.2 adds the
`internal_behaviors` section for these.

This migration is additive — you're documenting what already happens,
not changing behavior. Common cases:

**commit_changes (git hooks):**

```yaml
internal_behaviors:
  hook_execution:
    - description: "git pre-commit and commit-msg hooks run if installed"
      controllable_via_parameter: false
      policy_relevance: "hooks may modify or reject the commit"
```

**fetch_url (DNS caching):**

```yaml
internal_behaviors:
  cache_lookup:
    - description: "DNS lookups cached at OS level"
      controllable_via_parameter: false
      policy_relevance: "DNS cache poisoning is out of scope"
  retry:
    - description: "No automatic retry on transient failure"
      controllable_via_parameter: false
      policy_relevance: "Caller must retry; intent is single-attempt"
```

When in doubt, document side behaviors. They appear in audit logs
(so policy auditors can see what happened) without becoming Cedar
policy concerns.

### S5. Capability purpose annotation (was F-027)

Every capability declaration in v0.2 needs `purpose:`. Three values:

- **user_facing**: directly serves the user's intent
- **implementation_detail**: required by the implementation but not
  conceptually part of the user's request
- **policy_required**: required because policy mandates it (audit
  log writes, etc.)

For migration, most capabilities are `user_facing`. Exceptions:

**write_file atomic mode** — the parent-directory write needed for
temp-file creation is `implementation_detail`:

```yaml
capability:
  filesystem:
    - operation: write
      paths: ["{path}"]
      purpose: user_facing
      # ...
    - operation: write
      paths: ["{path.parent}"]
      purpose: implementation_detail
      required_if: "atomic == true or create_parents == true"
      # ...
```

**commit_changes filesystem ops** — reading and writing `.git`
internals is `implementation_detail`; the user-facing capability is
the state_mutation:

```yaml
capability:
  filesystem:
    - operation: read
      paths: ["{repository_path}/.git", "{repository_path}"]
      purpose: implementation_detail
      recursive: true
      # ...
    - operation: write
      paths: ["{repository_path}/.git"]
      purpose: implementation_detail
      recursive: true
      # ...
  state_mutation:
    - # ... user_facing
```

This distinction lets Cedar policies focus on user-facing capabilities;
implementation_detail capabilities can be granted broadly without
operators having to enumerate them.

### S6. Working directory declaration (was F-012)

v0.1 templates used `cd <dir> && <tool> ...` patterns. v0.2 forbids
this in templates; the `working_directory` field declares the directory
context.

Migration example (commit_changes):

```yaml
# v0.1
implementation:
  - tool: git
    template: |
      cd {repository_path} && 
      git commit --message {message[shell_quoted]}

# v0.2
implementation:
  - tool: git
    working_directory: "{repository_path}"
    template: |
      git commit --message {message|shell_quoted}
```

The resolver implements `working_directory` using tool-native
facilities (`git -C`, `tar -C`, `make -C`) or process spawn options
(setting `cwd`). Spec authors don't need to know which mechanism is
used — they declare the desired directory and the resolver picks the
right approach.

Specs without `cd` patterns don't need this migration.

### S7. Conditional cases for templates (was F-022)

v0.1 used a flat parameter_mapping namespace for multi-discriminator
templates (e.g., `atomic_replace`, `nonatomic_append`). v0.2 uses
`cases:` with `when:` clauses.

Migration example (write_file):

```yaml
# v0.1
parameter_mapping:
  write_command:
    atomic_replace: |
      tmpfile=$(mktemp ...) && printf ... > "$tmpfile" && mv ...
    atomic_append: |
      printf ... >> {path}
    atomic_create_new: |
      ( set -C; printf ... > {path} )
    nonatomic_replace: |
      printf ... > {path}
    nonatomic_append: |
      printf ... >> {path}
    nonatomic_create_new: |
      ( set -C; printf ... > {path} )

# v0.2
template:
  cases:
    - when: "atomic == true and mode == replace"
      command: |
        tmpfile=$(mktemp ...) && printf '%s' {content|shell_quoted} > "$tmpfile" && mv -- "$tmpfile" {path}
    - when: "atomic == true and mode == append"
      command: |
        printf '%s' {content|shell_quoted} >> {path}
    - when: "atomic == true and mode == create_new"
      command: |
        ( set -C; printf '%s' {content|shell_quoted} > {path} )
    - when: "atomic == false and mode == replace"
      command: |
        printf '%s' {content|shell_quoted} > {path}
    - when: "atomic == false and mode == append"
      command: |
        printf '%s' {content|shell_quoted} >> {path}
    - when: "atomic == false and mode == create_new"
      command: |
        ( set -C; printf '%s' {content|shell_quoted} > {path} )
```

This is verbose but explicit. If you have a common template that grows
by single flags across cases, consider the `merge:` mode (Chapter
6.5.3) for less repetition.

For specs without multi-discriminator templates (most v0.1 specs),
this migration doesn't apply.

### S8. Tool registry references (was F-033)

v0.2 requires every tool referenced in `process.spawn` capabilities
and `implementation.tool` fields to exist in the Tool registry.

If your deployment doesn't have a Tool registry yet, create one as
part of the v0.2 migration:

```yaml
# registries/tools/v0.2/tools.lki.yaml
lki_version: "0.2"
tools:
  - name: git
    is_system_critical: false
    typical_path: "/usr/bin/git"
  - name: curl
    is_system_critical: false
    typical_path: "/usr/bin/curl"
  - name: cat
    is_system_critical: false
    typical_path: "/bin/cat"
  # ... etc for every tool referenced by your specs
```

`cedar-from-lki` validates that every tool referenced has a registry
entry. Missing entries produce errors at projection time.

The existing v0.1 specs reference these tools (across all seven):
- git (commit_changes)
- curl (fetch_url, primary), wget (fetch_url, alternate)
- rm (remove_files)
- find (list_directory, primary), ls (list_directory, alternate)
- ripgrep, grep (grep_files)
- cat (read_file)
- shell built-ins, mv, mkdir, dd (write_file)

The "shell built-ins" case is awkward — bash, sh, dash aren't tools
in the LKI sense. v0.2 specs should declare specific tool names rather
than relying on shell-built-in behavior. For write_file specifically,
the `printf` builtin and `mv` external should both be registered as
tools.

### S9. Streaming mode declaration (was F-016)

v0.2 adds explicit `streaming: bounded | paginated | streamed`
declaration on output types. Default is `bounded` (v0.1 behavior),
so this is opt-in only when migrating to a non-bounded mode.

For the v0.1 specs:
- All current intents are `bounded` mode (single result, max_size /
  max_entries cap)
- list_directory could be migrated to `paginated` for v2.1.0 (a minor
  bump after v2.0.0) once a pagination implementation exists
- grep_files could be migrated to `streamed` for v2.1.0 for very large
  codebases

For the v0.2 migration itself, leave all specs as `bounded`. Pagination
and streaming variants can be added later as separate intents per the
variant pattern (Chapter 1).

## Part III — Worked example: fetch_url migration

The full v0.1 → v0.2 migration of fetch_url, with side-by-side
comparison of the substantive changes.

### v0.1 form (frozen as historical)

```yaml
spec:
  intent: fetch_url
  description: Retrieve content from a URL via HTTP/HTTPS, optionally writing to a file.

  signature:
    inputs:
      - name: url
        type: URL
        constraint: "scheme must be http or https"
      - name: output
        type: FilePath
        optional: true
      - name: timeout
        type: Duration
        default: 30s
        constraint: "must be <= 300s"
      # ... other parameters

  capability:
    network:
      - direction: outbound
        host: "{url.host}"
        port: "{url.port}"
        protocol: "{url.scheme}"
    filesystem:
      - operation: write
        path: "{output}"
        required_if: "output is not null"

  # ... behavior, constraints, examples, anti_patterns
```

### v0.2 form (the migrated version)

```yaml
lki_version: "0.2"

spec:
  intent: fetch_url
  intent_version: "2.0.0"
  description: Retrieve content from a URL via HTTP/HTTPS, optionally writing to a file.

  signature:
    inputs:
      - name: url
        type: URL@1.x
        constraint: "scheme must be http or https"
      - name: output
        type: FilePath@1.x
        optional: true
      - name: timeout
        type: Duration@1.x
        default: 30s
        constraint: "must be <= 300s"
        is_resource_bound: true       # NEW (M3.1 / S1)
      # ... other parameters

  behavior: |
    # ... unchanged ...

  constraints:
    parameter_constraints: []         # CHANGED (M3): split structure
    cross_parameter_constraints:
      - description: "HEAD method incompatible with output destination"
        rule: "method == HEAD implies output is null"

  capability:
    network:
      - direction: outbound
        host: "{url.host}"
        port: "{url.port}"
        protocol: "{url.scheme}"
        purpose: user_facing          # NEW (S5)
    filesystem:
      - operation: write
        paths: ["{output}"]           # CHANGED: was scalar 'path', now list
        purpose: user_facing          # NEW (S5)
        required_if: "output is not null"

  resolver_constraints: {}             # NEW (S2): empty for fetch_url

  internal_behaviors:                  # NEW (S4)
    cache_lookup:
      - description: "DNS lookups cached at OS level"
        controllable_via_parameter: false
        policy_relevance: "DNS cache poisoning is out of scope"
    retry:
      - description: "No automatic retry on transient failure"
        controllable_via_parameter: false
        policy_relevance: "Caller must retry; intent is single-attempt"

  implementation:
    - tool: curl
      priority: 100                    # NEW: explicit priority
      requires: "curl >= 7.50"
      tested_against: ["curl 7.x", "curl 8.x"]   # CHANGED (M6): condensed to ranges
      template: |
        curl --fail --silent --show-error
             --location={follow_redirects}
             --max-time {timeout.seconds}
             {header_flags} {method_flag} {output_flag}
             {url}
    - tool: wget
      priority: 50
      requires: "wget >= 1.20"
      # ... etc

  # ... examples and anti_patterns unchanged ...
```

### Differences summary for fetch_url

| Change | Reason | Section |
|--------|--------|---------|
| `lki_version: "0.2"` | Format upgrade | M1 |
| `intent_version: "2.0.0"` | Format major bump | M2 |
| Type references `@1.x` | Type version pinning | M4 |
| `constraints` split into two subsections | Structure clarity | M3 |
| `is_resource_bound: true` on `timeout` | Resource bound annotation | S1 |
| `purpose: user_facing` on all capabilities | Required in v0.2 | S5 |
| `resolver_constraints: {}` (empty) | Required section, no resolver-side checks | S2 |
| `internal_behaviors:` documented | DNS cache, retry policy | S4 |
| `implementation.priority` added | Required in v0.2 for multi-impl | M2/implicit |
| `tested_against` range syntax | Cleaner version expression | M6 |

No state_mutation in fetch_url so S3 doesn't apply. No multi-discriminator
templates so S7 doesn't apply. No `cd && ...` patterns so S6 doesn't
apply. fetch_url is one of the lighter migrations among the seven v0.1
specs.

## Part IV — Migration checklist

A checklist for migrating any v0.1 spec to v0.2. Work through these
in order.

### Phase 1: mechanical (lki-migrate handles this when the tool exists)

- [ ] Set `lki_version: "0.2"` at top
- [ ] Bump `intent_version` to next major (1.x.x → 2.0.0)
- [ ] Add `@1.x` pins to all non-primitive type references
- [ ] Restructure `constraints:` into `parameter_constraints` and
      `cross_parameter_constraints` subsections
- [ ] If `implementation:` has multiple entries, add explicit `priority:`
      fields
- [ ] Optionally condense `tested_against` exact lists to ranges
- [ ] Verify `examples:` structure unchanged
- [ ] Verify `anti_patterns:` structure unchanged

### Phase 2: substantive — capability section

- [ ] Add `purpose:` to every capability declaration (`user_facing` is
      the typical default)
- [ ] Identify implementation-detail capabilities (parent-directory
      writes for atomic ops, git internals reads for commit, etc.) and
      mark `purpose: implementation_detail`
- [ ] Extract resolver-side checks from `capability:` into
      `resolver_constraints:` (regex complexity, encoding validation,
      shell metacharacter rejection, etc.)
- [ ] For filesystem capabilities with `recursive: true`, add
      `recursion_safety:` block

### Phase 3: substantive — state mutation

- [ ] For each `state_mutation:` declaration:
    - [ ] Replace v0.1's `resource: "<class>:<identifier>"` string with
          structured `resource_type: <EntityType>@<version>` and
          `resource_identifier: <interpolation>`
    - [ ] Determine the operation's actual `visibility:` (immediate /
          rename_atomic / transactional)
    - [ ] Determine the operation's actual `durability:` (in_memory /
          committed / fsynced / fsynced_with_dir)
    - [ ] Determine the operation's actual `reversibility:` (irreversible /
          reversible_locally / reversible_until_remote_sync /
          undo_via_compensating_action)
    - [ ] Add `purpose:` (typically user_facing for state mutations)

### Phase 4: substantive — resource bounds

- [ ] Identify parameters that cap resource use (timeout, max_size,
      max_entries, max_count, page_size, etc.)
- [ ] Add `is_resource_bound: true` annotation to each

### Phase 5: substantive — internal behaviors

- [ ] Identify side behaviors the intent performs internally (hooks,
      retries, cache lookups, transient sub-process spawning)
- [ ] Add `internal_behaviors:` section documenting each
- [ ] Note `controllable_via_parameter:` and `policy_relevance:` for each

### Phase 6: substantive — implementation

- [ ] Replace any `cd <dir> && <tool> ...` patterns in templates with
      `working_directory:` field plus a template that doesn't `cd`
- [ ] For multi-discriminator templates, convert flat parameter_mapping
      namespace to `cases:` with `when:` clauses
- [ ] Apply pipe-syntax render modifiers (`{name|shell_quoted}`) to
      replace any ad-hoc quoting notation

### Phase 7: registry validation

- [ ] Verify every `tool:` reference in implementations has a
      corresponding Tool registry entry
- [ ] Verify every `resource_type:` in state_mutation/access_grant has
      a corresponding entity type in the type registry
- [ ] Verify every shared type reference has the expected version
      available in the type registry

### Phase 8: verification

- [ ] Run `lki-validate` against the migrated spec (when the tool
      exists; manual review until then)
- [ ] Run `cedar-from-lki` to produce schema and policy artifacts;
      verify they're sensible
- [ ] Re-run audit log examples against the new structure to confirm
      compatibility
- [ ] Verify intent variants share Cedar actions appropriately
      (Chapter 9.10)

## Part V — Migration policy

### Migrating one spec at a time vs all at once

Migrate one spec at a time. Each migration surfaces format questions
that can shape subsequent migrations. Migrating in parallel risks
applying inconsistent interpretations to similar questions across
specs.

Suggested order (lightest to heaviest):
1. fetch_url (no state mutation, no multi-discriminator)
2. read_file (similar shape to fetch_url)
3. grep_files (filesystem read with resolver_constraints extraction)
4. list_directory (filesystem read with recursion_safety)
5. remove_files (filesystem delete; state_mutation but simple)
6. commit_changes (state_mutation with non-trivial three-axis decision;
   working_directory required)
7. write_file (state_mutation with parameter-dependent three-axis;
   multi-discriminator template)

This order builds confidence — the early migrations are mostly
mechanical with some annotation work; the later ones require more
judgment.

### What to do when migration reveals format issues

Sharp edges discovered during migration are exactly the findings
that should be recorded against v0.2. Add them to FINDINGS_v0.2.md
with `[migration]` as the source tag.

Don't try to fix the format during migration. Record the issue and
continue. v0.3 will address accumulated migration findings the same
way v0.2 addressed v0.1 findings.

### Preserving v0.1 specs

Migrated v0.1 specs are *not* deleted. They remain in the versioned
spec directory as the v1.0.0 entries:

```
specs/cli/fetch_url/
  v1.0.0.yaml   # original v0.1 form, frozen
  v2.0.0.yaml   # v0.2 migration
```

A spec consumer can pin to `fetch_url@1.x` to use the v0.1 form
indefinitely. The migration provides v0.2 as an upgrade path, not a
forced replacement.

### Cedar projection artifacts

When a spec migrates to a new intent version, its cedar projection
artifacts also migrate:

```
projections/fetch_url/
  v1.0.0/
    schema.cedarschema
    policies.cedar
    pdp-query.yaml
  v2.0.0/
    schema.cedarschema      # re-projected per v0.2 rules
    policies.cedar          # regenerated three-tier
    pdp-query.yaml          # new pipeline structure
```

Re-projection is mechanical once the spec migration is complete.
`cedar-from-lki` processes the v0.2 spec and produces the v2.0.0
artifacts.

## What this document doesn't cover

- **lki-migrate tool internals.** Tool implementation is separate from
  format specification. When the tool exists, its behavior follows the
  rules in this document.
- **Migrating specs from formats earlier than v0.1.** v0.1 was the
  first format version; there is no pre-v0.1 form.
- **Migrating cross-version (skipping a version).** Migration is
  always to the immediately-next format version. v0.1 to v0.3 would
  go through v0.2.
- **Reverse migration.** v0.2 specs cannot be downgraded to v0.1; they
  use features that v0.1 doesn't support.

## Closing this guide

This document closes when v0.3 of the format spec ships. At that
point, the guide is renamed `MIGRATION_v0.1_to_v0.2.md` and archived
in the versioned record. A new `MIGRATION_FROM_v0.2.md` opens for

The migration discipline is part of the format evolution discipline
(Chapter 15 of LKI_FORMAT_v0.2.md). Each format revision produces
both the new format spec and the migration guide; together they form
the upgrade path.
